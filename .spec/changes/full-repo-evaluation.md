## Assessment

### Overall

This repo is coherent and usable. The core workflow is understandable, the local hooks are backed by real tests, the bootstrap context is under budget, and the repo has a clear point of view: local-first, zero-dependency, markdown plus bash.

The main issue is not conceptual failure. It is drift. The repo now has multiple sources of truth for startup behavior, review commands, scaffolding, and visualization. The workflow core is still solid, but the supporting surfaces are starting to disagree with each other and the viewer stack is consuming disproportionate complexity for a project whose main promise is minimalism.

### What is working

- Workflow core is strong. `.spec/FLOW.md`, `.spec/b-startup.md`, the hook chain, and `tests/test-spec-gate.sh` form a coherent system with a clear state machine and real enforcement.
- The local/CI story is clean. `.github/workflows/validate.yml` runs `setup.sh`, checks SOD freshness, and runs the hook tests.
- The context budget work paid off. `.spec/FLOW.md` is materially leaner than the earlier archived state, and the bootstrap target is currently under budget.
- The repo has a real product shape. `README.md`, `docs/index.html`, `LICENSE`, `packs/`, and `.github/` make it publishable rather than just a personal prompt scaffold.

### Findings

1. High: `setup.sh` ships a stale change-file template. The live template includes `files:` metadata for the scope gate (`.spec/changes/_template.md:1-3`), but the template generated by `setup.sh` omits it (`setup.sh:27-65`). Downstream repos bootstrapped from `setup.sh` get scaffolding that lags the current workflow contract.

2. High: Claude-specific review instructions are stale. `CLAUDE.md:9` still tells Claude users to call Codex with `codex -q`, while `.spec/FLOW.md:84-86` says Claude should use `codex exec`. That is direct protocol drift in a repo whose process depends on peer review actually working.

3. High: `scripts/build-viewer.sh` parses archived change filenames in the wrong order. The generic `YYYY-MM-DD-name` regex at `scripts/build-viewer.sh:62-64` matches before the more specific `YYYY-MM-DD-HHMMSS-name` pattern at `:65-67`, so archived files like `2026-04-15-005156-merge-and-advance-workflow.md` are misnamed in the viewer and lose the intended timestamp precision.

4. Medium: the JSONL append helpers do not fully satisfy their own “safe JSON” promise. `scripts/devlog.sh:25-30` and `scripts/flowlog.sh:29-34` only normalize newline, tab, and carriage return. Other control characters are neither escaped nor rejected, so invalid JSON can still be emitted from unusual input.

5. Medium: the viewer stack is now the dominant complexity center of the repo. `.spec/sod-report.md:54` shows `docs/viewer.html` at 20,461 estimated tokens, roughly a third of repo token weight by itself; `scripts/build-viewer.sh:1-220` adds another sizable bespoke parser/rewriter. For `spec-of-dust`, this is the clearest mismatch between product ethos and current implementation weight.

6. Medium: archive hygiene is not being exercised consistently. `.spec/FLOW.md:52` says finished change files move to `.spec/archive/` after merge, but `.spec/sod-report.md:35-43` still lists many completed real change files in `.spec/changes/`. The rule exists, the helper exists, but the repo state suggests the archival step is not reliably happening in normal use.

7. Advisory: repo guidance still has duplicated startup and workflow instructions. `README.md`, `AGENTS.md`, `CLAUDE.md`, `CODEX.md`, `.spec/b-startup.md`, and `.spec/FLOW.md` are better than they were, but they still overlap enough that drift is now appearing in practice. The `CLAUDE.md` command mismatch is the concrete symptom.

8. Advisory: viewer robustness depends on silent failure. `docs/viewer.html:591-606` swallows parse errors for embedded log and change data. That keeps the UI resilient, but it also hides corruption in generated viewer state, which is exactly the sort of silent drift this repo is supposed to prevent.

### Surface-by-surface evaluation

#### Workflow core: `.spec/`, `.githooks/`, tests

- `.spec/FLOW.md` earns its place. It is the real contract and, after trimming, is compact enough to justify itself.
- `.spec/b-startup.md` is appropriately small and useful.
- `.githooks/_spec_gate.sh` is large, but it earns the size because it centralizes the actual policy and is covered by `tests/test-spec-gate.sh`.
- `tests/test-spec-gate.sh` earns its place. For a policy-heavy repo, the hook tests are one of the strongest signals of seriousness.
- `scripts/merge-completed-work.sh` and `scripts/archive-done-changes.sh` are reasonable, but the repo’s current state suggests the archival path is underused or too easy to bypass.

#### Core docs: `README.md`, `AGENTS.md`, `CLAUDE.md`, `CODEX.md`

- `README.md` is useful and largely earns its length because this repo is also a public product surface.
- `AGENTS.md` is concise and good.
- `CODEX.md` is aligned enough with the flow.
- `CLAUDE.md` does not currently earn trust because its peer-review command is stale.
- The main documentation issue is not excess length anymore; it is duplicated operational guidance across multiple files with weak synchronization.

#### Scripts

- `scripts/update-sod-report.sh` is heavy but justified. The self-measurement idea is central to this repo and the script is the enforcement backbone for that.
- `scripts/devlog.sh` and `scripts/flowlog.sh` are good ergonomics improvements, but they need stricter JSON escaping or input rejection.
- `scripts/build-viewer.sh` is the least aligned with the repo’s “dust” principle. It is doing markdown parsing, archive-name heuristics, data embedding, and HTML rewriting in bash. It works, but it is the most obvious condensation candidate.

#### Docs surfaces: `docs/`

- `docs/index.html` earns its place as a minimal publishable landing page.
- `docs/README.md` is fine and lightweight.
- `docs/viewer.html` is polished and genuinely useful, but it is now a product within the product. It changed the repo’s center of gravity.

#### Packs and publishing scaffolding

- `packs/index.json`, `packs/javascript/v0/README.md`, `.github/README.md`, and `LICENSE` are all lightweight and acceptable.
- `packs/` is intentionally skeletal. That is fine as long as it stays skeletal and does not become a second speculative subsystem before the workflow core stabilizes.

### Redundancy and overbuild

- The biggest redundancy is instruction duplication across `README.md`, `AGENTS.md`, `CLAUDE.md`, `CODEX.md`, and `.spec/FLOW.md`.
- The biggest overbuild vector is the viewer pipeline: embedded logs, embedded change history, a builder script, and repo-state visualization for a framework whose main value is otherwise plain text and small bash hooks.
- The second overbuild vector is keeping completed changes both as active nearby context and as a growing historical product surface. Once many `done` files remain in `.spec/changes/`, the “active work” directory stops feeling active.

### Recommended next actions

1. Fix source-of-truth drift first. Make `setup.sh` generate the current template shape, align `CLAUDE.md` with `.spec/FLOW.md`, and add a tiny test that asserts scaffolded files match the live template/features.
2. Fix the viewer history bug next. Correct the archive filename parsing order in `scripts/build-viewer.sh` and add one regression test or fixture check for `YYYY-MM-DD-HHMMSS-name.md`.
3. Decide whether the viewer is core or optional. If it is core, test it and accept the complexity. If it is optional, cut scope now: the first thing to drop is embedded change-history parsing.
4. Tighten JSON helper correctness. Either escape all control characters in `devlog.sh` and `flowlog.sh` or reject unsupported bytes explicitly.
5. Restore archive discipline. Use the archive helper consistently so `.spec/changes/` reflects active work rather than becoming a second archive.
6. Consolidate duplicated operator guidance. Keep the normative protocol in `.spec/FLOW.md` and reduce the agent-specific files to short adapter notes.

### Recommendation

Do not restructure the whole repo before proceeding, but do trim and repair the drift before adding more meta-features.

Concretely: proceed only after fixing the three high-priority issues above (`setup.sh` template drift, `CLAUDE.md` review-command drift, and `build-viewer.sh` archive-name parsing). After that, pause on new workflow surface area and choose whether the viewer is a first-class product feature or something to condense. The workflow core is good enough to continue using; the surrounding tooling needs tightening before more layers are added.
