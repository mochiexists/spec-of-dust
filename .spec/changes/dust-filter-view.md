# Add dust filter view to show only the poetic closure lines

## What
The viewer already renders the "Dust" closure line for each completed change as a styled italic line, but it's mixed in with all the other filter views (All, Devlog, Flowlog, Changes). Add a "Dust" filter that shows *only* the dust lines from completed changes — a compact, gallery-style view where the poetic one-liners are the focus, not the metadata around them. This makes the dust itself browsable as a body of work.

## Acceptance criteria
- [ ] `templates/dust.html` adds a new filter button `<button class="filter-btn" data-filter="dust">Dust</button>` to the toolbar, positioned last after "Changes"
- [ ] Membership: Dust view includes only entries where `_source === "change"` AND `status === "done"` AND the dust field passes a shared `hasRealDust(entry)` helper (non-empty, not `nothing notable`, not just the `- Dust:` label). Devlog, flowlog, and non-done change entries are all hidden.
- [ ] A shared helper function `hasRealDust(entry)` is added at module scope and used by BOTH the filter membership check AND the existing `buildChangeEntry` dust-line rendering, so they cannot diverge
- [ ] Each shown entry renders in a minimal form: just the dust line, the change name, and the date. Closure metadata (challenges/learnings/outcomes), peer reviews, and verify sections are hidden in this view.
- [ ] The "Active Work" section at top is hidden when Dust filter is active (consistent with status=done filter)
- [ ] The "X shown" count reflects the number of visible Dust cards, not the raw `change`-sourced count
- [ ] Both `templates/dust.html` (source-of-truth for fresh setups) AND `docs/dust.html` (this repo's instance) receive the same HTML/JS changes. `build-dust.sh` only rewrites the embedded-data block, so template structural changes must also be applied to `docs/dust.html` directly. `bash scripts/build-dust.sh --check` passes after the change (data block consistent).

## Notes
- The existing inline `isReal()` check in `buildChangeEntry` guards the dust-line rendering today. Extract to module-scope `hasRealDust(entry)` so filter membership and rendering share one source of truth.
- No new CSS needed — the `.dust-line` class already has the styling. We just omit the other fields when rendering in dust mode.
- This is UI-only: `templates/dust.html` for source-of-truth, `docs/dust.html` auto-regenerated via `scripts/build-dust.sh`. No script logic changes, no gate changes, no b-startup changes.
- Keep it simple: no sorting controls, no search. Just the dust, chronologically (same order as the main view).
