# Add SOD analysis and versioning

## What
Add a lightweight `sod` reporting flow to `spec-of-dust` so the repo tracks its own footprint over time. Generate a committed analysis file with per-file lines, words, characters, and estimated tokens; expose a short summary in `README.md`; require commits that change the repo to include the refreshed analysis file; and start explicit project versioning at `0.0.1`.

## Acceptance criteria
- [ ] A committed SOD analysis file exists at `.spec/sod-report.md` and is generated by a repo-local script at `scripts/update-sod-report.sh`
- [ ] The SOD report covers Git-tracked text files only and records per-file lines, words, characters, and estimated tokens using `ceil(characters / 4)`
- [ ] `README.md` includes a short generated SOD summary with current totals and the current version from `VERSION`
- [ ] The hook flow blocks commits that change tracked text files in this repo unless the refreshed SOD report is staged; trivial skip commits remain governed by the existing skip path
- [ ] A project version file exists at `VERSION`, starts at `0.0.1`, and the docs describe the lightweight manual versioning convention

## Notes
- Keep it zero-dependency: shell plus standard Unix tools only
- Use a deterministic estimate for tokens: `ceil(characters / 4)`
- Keep the README summary short; detailed output should live in the generated analysis file
- Use `sod` as the short name in docs and generated output
- Scope the report to Git-tracked text files and exclude the generated report file itself from the measured totals
- Version bumps are manual and lightweight, not automatic on every refresh
